# CSGClaw Agent Teams Beta 1

Agent Teams Beta 1 的目标很简单：用户给一个 Team 一个目标，manager 自动拆成子任务，系统把子任务派给合适的 worker，worker 执行并回报结果，最终主任务自动汇总完成。

这不是要做一个新的聊天工具，而是把 Agent 协作从“靠聊天里口头派活”升级成“聊天里可见、系统里有状态、任务能自动流转”的协作方式。

---

## 1. 一句话理解

以前的多 Agent 协作更像这样：

```text
manager 在群里 @worker 说一句要做什么
worker 看懂了就去做
做完再回群里说结果
manager 自己记住谁做完了、谁还没做、下一步该给谁
```

Agent Teams 希望变成这样：

```text
用户提出目标
CSGClaw 记录成一个主任务
manager 拆成多个子任务
系统按顺序派给 worker
worker 领取、执行、回报
系统自动推进后续任务
所有子任务完成后，主任务自动汇总完成
```

核心变化是：任务进度不再只靠聊天记录和 Agent 自觉记忆，而是由 CSGClaw 持续记录和推进。

## 2. 主要概念

### 2.1 Team

Team 是一个可复用的小队。一个 Team 里通常包含：

- 一个 manager：负责理解目标、拆任务、协调执行。
- 多个 worker：负责真正执行某类任务。
- 一个 Team 房间：用于小队创建、成员协作和任务入口。

用户可以把 Team 理解成“一个有队长和队员的 Agent 工作组”。

### 2.2 Manager

Manager 是 Team 的负责人。它不会直接把用户目标一股脑丢给某个 worker，而是先判断：

- 这个目标要不要拆成多个步骤。
- 哪些步骤可以并行。
- 哪些步骤必须等前一步完成后才能做。
- 哪个 worker 更适合做哪个子任务。
- 每个子任务应该交付什么结果。

### 2.3 Worker

Worker 是执行者。它收到明确派发后，需要先领取任务，再执行任务，最后回报结果。

这里的“领取”很重要：它表示 worker 已经正式接手这个子任务，系统也会把这个任务标记为进行中，避免多个任务或多个 worker 混在一起。

### 2.4 主任务和子任务

主任务是用户真正想完成的目标，例如：

```text
帮我做一个前后端都完整的 Agent Teams 任务看板
```

子任务是 manager 拆出来的可执行工作，例如：

```text
1. 后端梳理任务状态和事件模型
2. 前端实现任务看板和详情时间线
3. 后端补充任务派发和完成逻辑
4. 前端接入真实数据并验证体验
```

用户通常关心主任务是否完成；系统和 worker 需要关心每个子任务的状态。

### 2.5 执行房间

Team 有自己的小队房间，但每个主任务也会有一个执行房间。

执行房间的作用是把一个任务的过程集中起来：

- 系统在这里通知 worker 有任务可做。
- worker 在这里领取任务、报告进度、提交结果。
- 用户可以进入这里看这个任务的具体执行过程。

可以把它理解成“这个主任务的项目群”。

### 2.6 派发和领取

派发是系统告诉某个 worker：“这个子任务轮到你了。”

领取是 worker 回应系统：“我已经接手这个子任务。”

这两个动作分开，是为了避免一个常见问题：群里虽然提到了 worker，但 worker 未必真正开始做。Beta 1 会把“已派发”和“已领取/进行中”区分开。

### 2.7 状态和时间线

每个任务都有状态。用户可以通过看板看到任务现在处在哪一步，也可以通过详情页看到任务发生过什么。

房间里的消息是给人看的过程记录；任务状态是系统用来推进流程的依据。也就是说，聊天记录让过程可见，任务状态让流程可靠。

这也是它和普通聊天派活最大的区别：普通聊天只留下消息，Agent Teams 还会知道任务派给了谁、有没有领取、是否完成，以及下一步能不能继续。

### 2.8 审批和阻塞

有些任务执行到一半可能需要人类确认，例如是否允许改某个配置、是否继续执行某个风险操作。这个时候任务可以进入阻塞或审批状态，等人类处理后再继续。

Beta 1 已经有这类概念和部分流程，但完整审批 UI 还在后续补齐。

## 3. 功能层主流程图

这张图只描述用户能理解的功能流程，不描述代码实现。

```text
                         +--------------------+
                         |       用户          |
                         |  提出一个目标       |
                         +----------+---------+
                                    |
                                    v
                         +--------------------+
                         |       Team         |
                         | manager + workers  |
                         +----------+---------+
                                    |
                                    v
                         +--------------------+
                         |      Manager       |
                         | 理解目标并拆任务   |
                         +----------+---------+
                                    |
                    +---------------+---------------+
                    |                               |
                    v                               v
          +--------------------+          +--------------------+
          |      任务看板       |          |      执行房间       |
          | 主任务 + 子任务状态 |          | 派发、领取、回报    |
          +----------+---------+          +----------+---------+
                     |                               |
                     |                               v
                     |                    +--------------------+
                     |                    |      Worker        |
                     |                    | 领取并执行子任务   |
                     |                    +----------+---------+
                     |                               |
                     +---------------+---------------+
                                     |
                                     v
                          +---------------------+
                          |   CSGClaw 自动推进  |
                          | 记录状态、检查依赖  |
                          | 派发后续、汇总结果  |
                          +----------+----------+
                                     |
                                     v
                          +---------------------+
                          |       用户看到       |
                          | 进度、结果、阻塞点  |
                          +---------------------+
```

主流程可以按 7 步理解：

1. 用户选择一个 Team，并输入想完成的目标。
2. 系统创建一个主任务。
3. Manager 根据 Team 成员能力把主任务拆成子任务。
4. 系统为这个主任务准备执行房间。
5. 系统把当前可以做的子任务派发给对应 worker。
6. Worker 领取、执行、完成或说明阻塞。
7. 系统检查依赖，继续派发后续子任务；所有子任务完成后，主任务自动完成并汇总结果。

## 4. 用户在界面里会看到什么

### 4.1 Team 页面

Team 页面用于理解“这支小队是谁、能做什么”。

用户可以看到：

- Team 名称和状态。
- manager 是谁。
- 有哪些 worker。
- 这个 Team 下有哪些主任务。

当前版本里，Team 成员仍然主要来自房间成员。后续会把 Team 成员管理做得更独立、更完整。

### 4.2 Tasks 页面

Tasks 页面用于看任务进度。

左侧或列表里主要展示主任务；点进主任务后，右侧看的是它下面的子任务看板。

看板会按状态分组，例如：

```text
待开始 -> 已派发 -> 进行中 -> 阻塞 -> 已完成 -> 失败
```

用户不需要知道每个状态背后的技术细节，只需要知道：

- 待开始：任务还没有真正进入执行。
- 已派发：系统已经通知 worker，但 worker 可能还没正式接手。
- 进行中：worker 已经领取并开始做。
- 阻塞：worker 或流程遇到问题，需要处理。
- 已完成：任务已经交付结果。
- 失败：任务执行失败，需要重试或人工处理。

### 4.3 任务详情

任务详情用于回答三个问题：

- 这个任务要做什么。
- 当前是谁在做，做到哪一步了。
- 过程中发生过什么。

详情里会展示任务描述、负责人、领取人、所属 Team、所属房间、派发时间、依赖、结果、错误等信息。当前工作区也在补活动时间线，让用户能更直观看到任务从创建、规划、派发、领取到完成的过程。

### 4.4 执行房间

执行房间用于看具体协作过程。

用户会在这里看到：

- 系统提示某个任务已经开始。
- 系统正式提醒某个 worker 领取任务。
- worker 的领取、完成、阻塞、失败反馈。
- 审批请求或人工介入信息。

执行房间不是替代任务看板，而是补充任务看板：看板适合扫状态，房间适合看上下文。

## 5. 当前 Beta 1 已经做到什么

Beta 1 当前已经能跑通一条完整闭环：

```text
创建 Team
  -> 创建主任务
  -> Manager 自动规划子任务
  -> 创建执行房间
  -> 派发可执行子任务
  -> worker 被提醒
  -> worker 领取任务
  -> worker 完成任务
  -> 后续子任务继续派发
  -> 所有子任务完成后主任务自动完成
```

已经重点解决的问题：

- Manager 规划结果不够标准时，不会轻易打断整个流程。
- 派发 worker 时使用真实 mention，避免 worker 收不到任务。
- worker 可以领取指定任务，避免旧任务抢走新任务。
- 旧 worker 能力不足时，也有兼容方式继续回报任务。
- Web 看板能展示主任务和子任务，并在任务活跃时自动刷新。

---

## 6. 当前仍然是 Beta 的地方

Beta 1 是可运行闭环，不是最终形态。当前主要限制是：

1. Team 和房间还没有完全解耦。
   现在 Team 成员很大程度上仍依赖房间成员，后续需要更明确的 Team 成员管理。

2. 执行房间生命周期还不完整。
   当前可以创建和使用执行房间，但归档、清理、重命名、重新绑定等能力还需要补。

3. 审批体验还不完整。
   审批相关概念已经存在，但完整 UI 和更顺手的人类确认流程还需要继续做。

4. worker 的超时、重试和恢复还需要增强。
   例如 worker 长时间不响应、任务派发后无人领取、旧派发需要恢复等。

5. Web 上的状态表达还需要继续打磨。
   特别是阻塞、审批中、失败这几类状态，需要让用户一眼看懂差异。

6. 老版本 worker 的能力升级还不够自动。
   已经运行中的 worker 可能仍然使用旧模板或旧命令，需要重建或补充兼容指令。

## 7. 使用时需要理解的几个细节

### 7.1 为什么要有“已派发”和“进行中”

已派发表示系统已经通知 worker；进行中表示 worker 已经正式领取。

这两个状态不能混在一起。因为 worker 被通知，不代表它已经开始做。区分这两个状态后，用户能看出问题到底是“还没轮到 worker”，还是“已经通知了但 worker 没响应”。

### 7.2 为什么要有执行房间

Team 房间适合管理小队，执行房间适合跟踪具体任务。

如果所有任务都挤在 Team 房间里，多个主任务的派发、结果、阻塞会混在一起。执行房间能把同一个主任务的上下文收在一起，方便用户和 Agent 追踪。

### 7.3 为什么 worker 要领取任务

领取是 worker 和系统之间的确认动作。

没有领取，系统只能知道“任务已经发出”；有了领取，系统才能知道“这个 worker 正在做这个任务”，并在看板里准确展示进行中。

### 7.4 为什么任务完成后能自动派发后续任务

Manager 规划时会告诉系统哪些任务依赖哪些任务。

例如：

```text
前端接入真实数据
  依赖于
后端先准备数据能力
```

当前置任务完成后，系统会发现后续任务可以开始，于是自动派发给对应 worker。

### 7.5 用户应该看哪里

- 想看整体进度：看 Tasks 看板。
- 想看一个任务发生了什么：看任务详情时间线。
- 想看 worker 和系统的具体交流：进执行房间。
- 想调整小队成员：看 Team 页面。

---

## 8. Beta 1 验收标准

这版功能至少应该满足：

1. 用户能创建 Team，并看到 Team 房间。
2. 用户能给 Team 创建主任务。
3. 主任务创建后，manager 能自动规划子任务。
4. 子任务能分配给合适 worker，并说明目标和交付物。
5. 系统能创建或进入这个主任务的执行房间。
6. 系统能把可执行子任务真实派发给 worker。
7. worker 能领取指定任务。
8. worker 完成后，任务状态能更新到已完成。
9. 有依赖的后续任务能在前置任务完成后继续派发。
10. 所有子任务完成后，主任务能自动完成并汇总结果。
11. 用户能在看板、详情和房间里看懂当前进度。
